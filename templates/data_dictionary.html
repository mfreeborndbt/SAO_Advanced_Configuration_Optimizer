{{ project_name }} / Data Dictionary

Definitions for all metrics and fields across the dashboard

Model Identity

TermDefinition
ModelThe name of a dbt model materialized as a table or incremental table in your data warehouse
SchemaThe database and schema where the model is materialized
LayerFirst-level folder classification from the dbt project FQN (e.g., staging, intermediate, marts)
Folder 1 / 2 / 3Directory hierarchy under the models/ directory in the dbt project
TypeMaterialization strategy: “table” (full rebuild each run) or “incremental” (appends/merges new data)
LanguageThe model’s language: SQL or Python
Tagsdbt tags assigned to the model via config or YAML
WarehouseThe Snowflake warehouse or Databricks compute cluster used to build the model. Orange = model-level override

SAO Configuration

TermDefinition
updates_onSAO freshness setting controlling rebuild triggers. “any” (default): rebuild if any upstream changed. “all”: only rebuild when all upstream dependencies have new data
build_afterSAO freshness setting defining the minimum time between rebuilds (e.g., “1 hour”, “1 day”). Prevents excessive rebuilding even when upstream data changes

Run Metrics (7-day window)

TermDefinition
RunsTotal number of successful model builds across all scheduled jobs in the 7-day analysis window
Max DailyThe highest number of runs this model had in any single day
JobsNumber of distinct scheduled jobs that built this model. >1 indicates job overlap (potential redundancy)
MJ1 / MJ2 / MJ3 NameName of the top 3 jobs (by run count for this model) that build this model. "MJ" = Model Job
MJ1 / MJ2 / MJ3 RunsNumber of times this model was built by each of the top 3 jobs
MJ1 / MJ2 / MJ3 SchedHuman-readable cron schedule for each job
MJ1 / MJ2 / MJ3 SAOWhether State-Aware Orchestration is enabled on each job (ON/OFF)
Env J1-J5Project-level top 5 jobs by total runs across all models. Shows how many times each job built this specific model. Hover the column header for job name and cadence

Execution Time

TermDefinition
Total TimeSum of execution times across all runs in the 7-day window
Avg TimeMean execution time per run
Min TimeFastest execution time observed
Max TimeSlowest execution time observed

Cost Estimates

TermDefinition
Total / Avg / Min / Max CostEstimated dollar costs computed as execution_time x cost_per_second. Cost per second is derived from the warehouse size mapping configured in the Warehouse Costs settings

Row Metrics

TermDefinition
Rows (start)Row count from the earliest run in the 7-day window
Rows (now)Row count from the most recent run
Row DeltaNet change in row count: Rows (now) minus Rows (start)
Runs with ChangeNumber of consecutive run pairs where the row count changed
Change %Percentage of run pairs that showed a row count change. 0% = completely static data
Avg New RowsEstimated average rows added per run (Row Delta / Total Runs)
ColumnsNumber of columns in the model’s catalog
ValuesTotal data points: Rows (now) x Columns

Lineage & Dependencies

TermDefinition
Upstream TablesCount of direct parent models materialized as table or incremental
Upstream Avg CostSum of upstream models’ average per-run costs
Downstream TablesCount of all transitive downstream models materialized as table or incremental (full BFS traversal)
Downstream Avg CostSum of downstream models’ average per-run costs. Represents the cascade cost triggered by rebuilding this model

Models Optimization Tab

TermDefinition
QualificationWhy the model was flagged: meets row/time thresholds (1M+ rows AND >1 min avg) or is in the top 20% most expensive per run. Models with any window functions (OVER clauses) are excluded
Primary KeyDetected via: (1) unique_key in dbt config, (2) contract enforcement, or (3) a column with both unique and not_null tests defined. The PK Columns field shows which columns were identified through tests
PK Columns (from tests)Columns that have both a unique test AND a not_null test, making them strong primary key candidates for incremental strategy
Has Date ColumnWhether any column name contains date-related patterns (date, timestamp, _at, created, updated, modified). Useful for incremental strategies based on date partitioning
Lines of CodeTotal line count of the model’s raw SQL file
JoinsNumber of JOIN clauses in the SQL
CTEsNumber of Common Table Expressions (WITH ... AS (...))
SubqueriesNested SELECT statements beyond the primary query
Window FunctionsCount of OVER() clauses (window/analytic functions)
CASE StatementsNumber of CASE WHEN expressions
GROUP BYsNumber of GROUP BY clauses
UNIONsNumber of UNION / UNION ALL operations
RefsCount of ref() macro calls (dbt model references)
SourcesCount of source() macro calls (external source references)
Jinja BlocksTotal Jinja template expressions ({{ '{{' }} {{ '}}' }} and {{ '{%' }} {{ '%}' }} blocks)

Jobs Optimization Tab

TermDefinition
ScheduleThe cron expression and human-readable cadence for the job
SAO EnabledWhether State-Aware Orchestration is enabled on this job
Total ModelsNumber of table/incremental models successfully built by this job
Shared ModelsModels in this job that are also built by at least one other job
Unique ModelsModels that are exclusively built by this job and no other
Redundancy %Percentage of the job’s models that overlap with other jobs: Shared / Total x 100
Execute StepsThe dbt CLI commands configured in the job (e.g., dbt build, dbt run --full-refresh)
Unique LogicSpecial operations that make this job non-consolidatable: full-refresh, seed, snapshot, source freshness, run-operation
Consolidation CandidateFlagged when: redundancy ≥ 50%, no unique logic, and similar or less frequent cadence compared to overlapping jobs
