Definitions for all metrics and fields across the dashboard

▶ Model Identity 8 fields
TermDefinition
ModelThe name of a dbt model materialized as a table or incremental table in your data warehouse
SchemaThe database and schema where the model is materialized
LayerFirst-level folder classification from the dbt project FQN (e.g., staging, intermediate, marts)
Root Folder / Subfolder 1 / Subfolder 2Directory hierarchy under the models/ directory in the dbt project
TypeMaterialization strategy: “table” (full rebuild each run) or “incremental” (appends/merges new data)
LanguageThe model’s language: SQL or Python
Tagsdbt tags assigned to the model via config or YAML
WarehouseThe Snowflake warehouse or Databricks compute cluster used to build the model. Orange = model-level override
▶ SAO Configuration 2 fields
TermDefinition
updates_onSAO freshness setting controlling rebuild triggers. “any” (default): rebuild if any upstream changed. “all”: only rebuild when all upstream dependencies have new data
build_afterSAO freshness setting defining the minimum time between rebuilds (e.g., “1 hour”, “1 day”). Prevents excessive rebuilding even when upstream data changes
▶ Run Metrics (7-day window) 8 fields
TermDefinition
RunsTotal number of successful model builds across all scheduled jobs in the 7-day analysis window
Max DailyThe highest number of runs this model had in any single day
JobsNumber of distinct scheduled jobs that built this model. >1 indicates job overlap (potential redundancy)
MJ1 / MJ2 / MJ3 NameName of the top 3 jobs (by run count for this model) that build this model. “MJ” = Model Job
MJ1 / MJ2 / MJ3 RunsNumber of times this model was built by each of the top 3 jobs
MJ1 / MJ2 / MJ3 SchedHuman-readable cron schedule for each job
MJ1 / MJ2 / MJ3 SAOWhether State-Aware Orchestration is enabled on each job (ON/OFF)
Env J1-J5Project-level top 5 jobs by total runs across all models. Shows how many times each job built this specific model
▶ Execution Time & Cost 5 fields
TermDefinition
Total / Avg / Min / Max TimeExecution time statistics across all runs in the 7-day window
Total / Avg / Min / Max CostEstimated dollar costs computed as execution_time × cost_per_second. Cost per second is derived from the warehouse size mapping configured in Settings
▶ Row Metrics 6 fields
TermDefinition
Rows (start / now)Row count from the earliest and most recent run in the 7-day window
Row DeltaNet change in row count: Rows (now) minus Rows (start)
Avg New RowsEstimated average rows added per run (Row Delta / Total Runs)
Runs with ChangeNumber of consecutive run pairs where the row count changed
ColumnsNumber of columns in the model’s catalog
ValuesTotal data points: Rows (now) × Columns
▶ Lineage & Dependencies 4 fields
TermDefinition
Upstream TablesCount of direct parent models materialized as table or incremental
Upstream Avg CostSum of upstream models’ average per-run costs
Downstream TablesCount of all transitive downstream models materialized as table or incremental (full BFS traversal)
Downstream Avg CostSum of downstream models’ average per-run costs. Represents the cascade cost triggered by rebuilding this model
▶ Incremental Readiness Signals 5 fields
TermDefinition
Has Primary TestThe detected primary key column(s) that have tests defined. Sources checked in priority order: (1) columns with both unique and not_null tests defined (queried directly from the Discovery API), (2) unique_key from dbt model config. Green = tests verified passing, blue = tests defined
Potential Primary KeyColumns detected by naming convention that may serve as primary keys. Matches column names containing _id, _key, _sk, or _pk patterns (e.g. user_id, account_key, dim_sk). Useful for identifying incremental key candidates when no formal tests exist
Date / TimeseriesWhether the model has a date or timeseries column suitable for incremental partitioning
% New RowsAverage new rows per run as a percentage of total rows (avg_new_rows / latest_rows × 100). Lower values = better incremental candidate
MacrosCustom (non-builtin) dbt macros detected in the model SQL. Models with custom macros may require macro changes for incremental conversion
▶ Complexity Signals 3 fields
TermDefinition
Window FunctionsCount of OVER() clauses (analytic functions). Models with window functions are harder to convert to incremental
GROUP BYsNumber of GROUP BY clauses
UNIONsNumber of UNION / UNION ALL operations
▶ Impact & Performance 5 fields
TermDefinition
RowsLatest row count from the most recent run
Avg Time / p95 TimeMean and 95th-percentile execution time per run. Large gaps between avg and p95 indicate high variance
Avg CostEstimated dollar cost per run based on warehouse size mapping
RunsNumber of times the model ran from scheduled jobs in the past week
DownstreamCount of all downstream table/incremental models. Higher = more cascade performance impact from optimizing this model
▶ Job Configuration & Performance 7 fields
TermDefinition
ScheduleThe cron expression and human-readable cadence for the job
Max DailyHighest number of runs this job had in a single day during the 7-day window
RunsTotal number of successful job runs in the 7-day window
Avg TimeMean job execution time (wall-clock duration) across all runs
p95 Time95th-percentile job execution time. Large gaps vs avg indicate high variance / skew
SAO EnabledWhether State-Aware Orchestration is enabled on this job
Execute StepsThe dbt CLI commands configured in the job (e.g., dbt build, dbt run --full-refresh)
▶ Overlap & Redundancy 6 fields
TermDefinition
Total ModelsNumber of table/incremental models successfully built by this job
Shared ModelsModels in this job that are also built by at least one other job
Unique ModelsModels that are exclusively built by this job and no other
Redundancy %Percentage of the job’s models that overlap with other jobs: Shared / Total × 100
Top Overlap 1 / 2 / 3The top 3 other jobs with the most model overlap, showing job name, shared model count, and overlap percentage
Unique LogicSpecial operations: full-refresh, seed, snapshot, source freshness, run-operation
